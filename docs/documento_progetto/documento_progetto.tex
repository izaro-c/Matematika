\section{Specifiche Informali}
Il progetto \textbf{Matematika} nasce dall'esigenza di modernizzare lo studio della geometria e della logica matematica. Tradizionalmente, lo studio dei teoremi avviene tramite la lettura di libri di testo in modo lineare e passivo. Questo approccio rende difficile comprendere come un concetto logico dipenda strettamente da un altro.

Per risolvere questo problema, è stata ideata e sviluppata un'applicazione web interattiva di tipo \textit{Single Page Application (SPA)}. Matematika rappresenta la conoscenza matematica non come un testo piatto, ma come un \textbf{grafo diretto aciclico (DAG)} in cui ogni nodo è un'entità (ad esempio, una Definizione, un Assioma o un Teorema). Di ogni nodo si vuole memorizzare l'identificativo, il tipo e le dipendenze dirette verso altri nodi. Ciascun Teorema può includere una o più Prove (dimostrazioni), dotate ciascuna di un proprio identificativo e delle proprie dipendenze logiche necessarie alla validità della dimostrazione.

Il sistema distingue due profili d'uso principali. Lo \textbf{Studente} è il fruitore dei contenuti matematici: esplora il grafo della conoscenza e interagisce con i diagrammi. L'\textbf{Autore} è il creatore dei contenuti: può aggiungere, modificare e rimuovere contenuti matematici tramite un Editor dedicato.

Le funzionalità e le caratteristiche principali del sistema includono:
\begin{itemize}
    \item \textbf{Pagina Principale e Rami della Matematica:} Lo Studente viene accolto da una panoramica generale che suddivide la materia nelle sue branche principali, facilitando l'orientamento all'interno del dominio di studio.
    \item \textbf{Esplorazione Non Lineare:} È possibile navigare liberamente tra i concetti tramite una mappa interattiva o un motore di ricerca rapido (Omnibar).
    \item \textbf{Glossario Interattivo e Marginalia:} Il sistema è dotato di un glossario dedicato per la consultazione rapida dei termini matematici. I significati sono accessibili sia tramite una pagina dedicata, sia cliccando direttamente su parole chiave collegate (link) o su formule matematiche, le quali aprono delle spiegazioni laterali (marginalia) senza interrompere il flusso di lettura.
    \item \textbf{Grafo degli Assiomi: Modalità Sandbox e Validazione in Tempo Reale:} Il cuore dell'applicazione è la rappresentazione delle dipendenze matematiche tramite un grafo interattivo. Tramite una speciale Modalità Sandbox, lo Studente può esplorare il grafo accendendo (abilitando) o spegnendo (disabilitando) a piacimento gli assiomi fondamentali. A ogni variazione, un motore di validazione in tempo reale ricalcola l'ordinamento topologico dell'intero sistema. I risultati si riflettono immediatamente sull'interfaccia: se un teorema perde il supporto logico delle sue premesse, viene visibilmente invalidato; se lo acquisisce, viene validato. In fase di inserimento di nuovi contenuti, il sistema verifica l'assenza di cicli nel grafo e, qualora rilevi una dipendenza ciclica, interrompe l'operazione di salvataggio emettendo un errore.
    \item \textbf{Dimostrazioni Dinamiche:} Le pagine di teoria contengono diagrammi geometrici generati dal motore JSXGraph. Tali grafiche reagiscono in tempo reale alle manipolazioni dell'utente.
    \item \textbf{Modelli Matematici e Sistemi Assiomatici:} Il sistema rappresenta modelli matematici concreti (istanze di sistemi assiomatici) e sistemi assiomatici astratti. Di ogni Modello e Sistema Assiomatico si vuole memorizzare l'identificativo, il titolo e l'elenco degli assiomi che lo compongono o che esso soddisfa.
    \item \textbf{Editor dei Contenuti:} L'Autore dispone di un ambiente di composizione integrato per aggiungere, modificare e rimuovere contenuti matematici (Teoremi, Definizioni, Assiomi, Dimostrazioni), definirne i metadati e stabilirne le dipendenze logiche con i nodi preesistenti del grafo.
\end{itemize}

In sintesi, Matematika mira a trasformare l'apprendimento della teoria da un esercizio di memoria a un'esperienza di scoperta logica, combinando il rigore dell'Ingegneria del Software con un'interfaccia utente moderna ed educativa.

\subsection{Revisione dei requisiti}
A seguito di una interazione (simulata) con gli stakeholder, la traccia è stata revisionata per chiarire ambiguità, disambiguare i termini, e separare le frasi che esprimono più requisiti. Di seguito riportiamo i requisiti revisionati, formulati in modo chiaro e verificabile.

\begin{enumerate}
    \item Il sistema deve offrire allo Studente una funzionalità per esplorare le branche della matematica.
    \item Di ogni Nodo si vuole memorizzare identificativo, tipo e dipendenze.
    \item Di ogni Prova si vuole memorizzare identificativo e dipendenze.
    \item Il sistema deve offrire allo Studente una funzionalità per navigare tra i concetti matematici in modo non lineare.
    \item Il sistema deve offrire allo Studente una funzionalità per consultare un glossario interattivo.
    \item Il sistema deve offrire allo Studente una funzionalità per modificare lo stato logico degli assiomi.
    \item Il sistema deve ricalcolare l'ordinamento topologico in base all'elenco dei nodi.
    \item Il sistema deve aggiornare la validità dei teoremi in tempo reale.
    \item Il sistema deve verificare l'assenza di cicli in fase di validazione build-time e durante l'inserimento di nuovi contenuti.
    \item Il sistema deve invalidare visivamente i nodi dipendenti che perdono le proprie premesse logiche.
    \item Il sistema deve validare i nodi dipendenti che le acquisiscono.
    \item Il sistema deve interrompere l'operazione di inserimento ed emettere un errore qualora rilevi una dipendenza ciclica.
    \item Il sistema deve rappresentare i modelli matematici e i sistemi assiomatici.
    \item Di ogni Modello e Sistema Assiomatico si vuole memorizzare identificativo, titolo e l'elenco di Assiomi.
    \item Il sistema deve offrire all'Autore una funzionalità per aggiungere contenuti matematici tramite un Editor.
    \item Il sistema deve offrire all'Autore una funzionalità per modificare contenuti matematici tramite un Editor.
    \item Il sistema deve offrire all'Autore una funzionalità per rimuovere contenuti matematici tramite un Editor.
    \item Il sistema deve generare diagrammi geometrici interattivi tramite il motore JSXGraph che reagiscono in tempo reale alle manipolazioni dell'utente.
\end{enumerate}

\subsection{Vincoli (Constraints)}
Nella generazione della Test Suite, per evitare un'esplosione combinatoria (3x3x2 = 18 combinazioni possibili) e rimuovere test irrilevanti o ridondanti, sono stati individuati i seguenti vincoli (\textit{Constraints}) tra le categorie:

\begin{itemize}
    \item \textbf{C1 [Error / Single]:} La categoria I2 (Nodi mancanti) rappresenta uno stato anomalo dell'ambiente. Viene testata con un'unica combinazione rappresentativa (A3) ignorando il tipo di dipendenza, per verificare la robustezza del ciclo di attraversamento.
    \item \textbf{C2 [If]:} La categoria D3 (Nessuna dipendenza) determina la validità a priori del nodo. Viene testata con A1 (Nessun assioma attivo) in modo da confermare che tali nodi diventino validi anche nel caso peggiore.

\end{itemize}

\section{Implementazione}
L'applicazione è stata sviluppata in ambiente Node.js utilizzando TypeScript, React 19 e Vite come bundler. La gestione dello stato è affidata a Zustand, lo styling a Tailwind CSS, e la validazione dei dati a Zod. I contenuti matematici sono scritti in formato MDX e compilati a build-time. In questa sezione vengono presentate le scelte implementative relative ai package dell'architettura BCED.

\subsection{Package Database}
Il package Database (\texttt{src/database/}) fornisce i dati all'applicazione. Poiché Matematika è una SPA statica, non utilizza un DBMS relazionale ma un database basato su file JSON pre-compilati e file MDX. Il ruolo di \textit{Data Access Object (DAO)} è svolto dalla classe \texttt{ContentStore} (\texttt{dao/content/ContentStore.ts}), che centralizza l'accesso ai contenuti e implementa il pattern Singleton. La classe mantiene in memoria mappe indicizzate per ciascun tipo di contenuto (teoremi, definizioni, assiomi, modelli, etc.) e li carica e valida tramite gli schemi Zod definiti in \texttt{schemas.ts}. Si riporta di seguito uno stralcio significativo della classe:

\begin{lstlisting}[language=Java]
export class ContentStore {
  // Mappe indicizzate per tipo di contenuto
  public theorems: Map<string, Theorem> = new Map();
  public definitions: Map<string, Definition> = new Map();
  public axioms: Map<string, Axiom> = new Map();
  public models: Map<string, Model> = new Map();
  // ... altre mappe per lessons, demos, mathematicians, etc.

  constructor() {
    this.init(); // Carica e valida tutti i contenuti all'avvio
  }

  private init(): void {
    // Itera sui loader MDX generati a build-time
    for (const loader of contentLoaders) {
      const { slug, frontmatter, default: Component } = loader;
      const entry = contentIndex[slug];
      if (!entry) continue;
      // Validazione del contenuto con lo schema Zod corrispondente
      const schema = this.getSchema(entry.contentType);
      if (schema) {
        const result = schema.safeParse(frontmatter);
        if (!result.success) continue; // Contenuto non valido, ignorato
      }
      // Archivia il contenuto validato nella mappa corrispondente
      this.store(entry.contentType, slug, frontmatter, Component);
    }
  }
}
\end{lstlisting}

La struttura dei dati persistenti è definita in due file JSON principali:
\begin{itemize}
    \item \texttt{contentIndex.json} — indice generato a build-time (\texttt{generate-content-index.ts}) che mappa ogni slug ai metadati del contenuto.
    \item \texttt{graph\_structure.json} — grafo delle dipendenze logiche con ordinamento topologico pre-calcolato, utilizzato dall'entità \texttt{Grafo} per la validazione in tempo reale.
\end{itemize}

\subsection{Package Entity}
Il package Entity (\texttt{src/entity/}) contiene le classi che rappresentano i concetti del dominio matematico e incapsulano la logica di business. È composto da tre file:

\begin{itemize}
    \item \texttt{Grafo.ts} — classe che incapsula la struttura del DAG. Fornisce il metodo \texttt{evaluate()} per la validazione topologica dei nodi, \texttt{isDagValid()} per la verifica di integrità strutturale, e il metodo statico \texttt{computeDisabledAxiomsFromModels()} per determinare quali assiomi disabilitare in base ai modelli matematici attivi.
    \item \texttt{Nodo.ts} — value object che rappresenta un singolo nodo del grafo. Espone le proprietà \texttt{isAxiom} e \texttt{isTheorem} e il metodo \texttt{isSatisfiedBy(validNodes)} che determina se il nodo è logicamente valido dato l'insieme corrente di nodi attivi, implementando la logica OR per i teoremi con dimostrazioni multiple.
    \item \texttt{graphTypes.ts} — interfacce TypeScript (\texttt{GraphStructure}, \texttt{GraphNodeMeta}, \texttt{GraphNodeProof}, \texttt{ModelInfo}, \texttt{SystemInfo}, \texttt{GraphEdge}) che definiscono i tipi del dominio.
\end{itemize}

Si riporta di seguito il metodo principale \texttt{evaluate()} della classe \texttt{Grafo}, responsabile del ricalcolo dell'ordinamento topologico e della validità dei nodi:

\begin{lstlisting}[language=Java]
export class Grafo {
  private readonly order: string[];          // Ordine topologico pre-calcolato
  private readonly nodi: ReadonlyMap<string, Nodo>; // Mappa dei nodi

  // Factory method: costruisce il grafo a partire dalla struttura JSON
  static from(structure: GraphStructure): Grafo {
    return new Grafo(structure);
  }

  // Valida la struttura DAG e restituisce i nodi logicamente validi
  evaluate(activeAxioms: Record<string, boolean>): Set<string> {
    const validNodes = new Set<string>();

    // 1. Inizializza i nodi validi con gli assiomi attivi
    for (const [axiomId, isActive] of Object.entries(activeAxioms)) {
      if (isActive) validNodes.add(axiomId);
    }

    // 2. Valuta i nodi seguendo l'ordine topologico
    for (const nodeId of this.order) {
      if (validNodes.has(nodeId)) continue; // Nodo già valido
      const nodo = this.nodi.get(nodeId);
      if (!nodo) continue;                  // Nodo assente nell'indice
      if (nodo.isAxiom) continue;           // Già gestito nel passo 1

      // Il nodo è valido se tutte le sue dipendenze sono soddisfatte
      if (nodo.isSatisfiedBy(validNodes)) {
        validNodes.add(nodeId);
      }
    }

    return validNodes;
  }
}
\end{lstlisting}

Il metodo itera sui nodi seguendo l'ordine topologico pre-calcolato, garantendo che le dipendenze di ciascun nodo siano state valutate prima del nodo stesso. Per i teoremi, la validità è determinata dal metodo \texttt{isSatisfiedBy()} della classe \texttt{Nodo}, che verifica se almeno una delle dimostrazioni ha tutte le proprie dipendenze soddisfatte (logica OR).

\subsection{Package Controller}
Il package Controller (\texttt{src/controller/}) contiene gli store Zustand che orchestrano la logica applicativa e coordinano i casi d'uso. Lo store principale per il caso d'uso UC6 è \texttt{GraphSandboxStore} (\texttt{store/GraphSandboxStore.ts}), che gestisce lo stato della Modalità Sandbox. Altri store rilevanti includono \texttt{GraphStore} per il grafo globale e \texttt{GlossaryStore} per il glossario.

\texttt{GraphSandboxStore} carica la struttura del grafo a partire da \texttt{graph\_structure.json} e istanzia l'entità \texttt{Grafo} tramite il factory method \texttt{Grafo.from()}. A ogni interazione dell'utente (toggle di un assioma), il Controller aggiorna lo stato degli assiomi attivi e delega la validazione all'entità \texttt{Grafo.evaluate()}. L'uso di Zustand garantisce che i componenti React siano reattivamente aggiornati a ogni cambiamento di stato, senza necessità di DTO intermedi.

\begin{lstlisting}[language=Java]
// Carica la struttura del grafo a build-time e istanzia l'entità Grafo
const staticGraphData = graphStructureData as GraphStructure;
const grafo = Grafo.from(staticGraphData);

export const useGraphSandboxStore = create<GraphSandboxState>((set) => ({
  activeAxioms: {},
  validNodes: new Set<string>(),

  // Metodo principale: attiva/disattiva un assioma e rivaluta il grafo
  toggleAxiom: (axiomId: string) =>
    set((state) => {
      const newAxioms = { ...state.activeAxioms };
      if (newAxioms[axiomId]) {
        delete newAxioms[axiomId];   // Disattiva l'assioma
      } else {
        newAxioms[axiomId] = true;   // Attiva l'assioma
      }
      // Delega la validazione all'entità Grafo
      const newValidNodes = grafo.evaluate(newAxioms);
      return { activeAxioms: newAxioms, validNodes: newValidNodes };
    }),

  // Ripristina lo stato iniziale: nessun assioma attivo
  clearSandbox: () => {
    const newValidNodes = grafo.evaluate({});
    set({ activeAxioms: {}, validNodes: newValidNodes });
  },
}));
\end{lstlisting}

Il metodo \texttt{toggleAxiom()} implementa il caso d'uso UC6: riceve l'identificativo dell'assioma dalla Boundary, inverte il suo stato logico (da acceso a spento o viceversa), e chiama \texttt{grafo.evaluate()} per ricalcolare la validità dell'intero grafo. Il metodo \texttt{clearSandbox()} ripristina lo stato iniziale valutando il grafo con un insieme vuoto di assiomi.

\subsection{Package Boundary}
Il package Boundary (\texttt{src/boundary/}) contiene i componenti React che gestiscono l'interfaccia utente. Il componente principale per il caso d'uso UC6 è \texttt{AxiomSandboxPanel} (\texttt{components/ui/AxiomSandboxPanel.tsx}), un pannello laterale che permette allo Studente di attivare o disattivare gli assiomi e di visualizzare in tempo reale lo stato di validità dei nodi.

Il componente ottiene l'accesso al Controller tramite l'hook Zustand \texttt{useGraphSandboxStore()}, da cui destruttura \texttt{activeAxioms}, \texttt{toggleAxiom}, \texttt{validNodes} e altri metodi. I dati degli assiomi e dei modelli sono caricati dal Database tramite il DAO \texttt{db} (\texttt{ContentStore}).

\begin{lstlisting}[language=Java]
export const AxiomSandboxPanel: React.FC = () => {
  // Accesso al Controller tramite hook Zustand
  const {
    activeAxioms, toggleAxiom, validNodes, clearSandbox
  } = useGraphSandboxStore();
  
  // Caricamento dati dal Database tramite il DAO ContentStore
  const models = db.getAllModels();
  const axioms = db.getAllAxioms();

  return (
    <div>
      {/* Renderizza una checkbox per ogni assioma */}
      {axioms.map(axiom => {
        const isActive = !!activeAxioms[axiom.id];
        return (
          <label key={axiom.id}>
            <input
              type="checkbox"
              checked={isActive}
              // Al click, invoca il Controller per il toggle
              onChange={() => toggleAxiom(axiom.id)}
            />
            {axiom.title}
          </label>
        );
      })}
      {/* Footer con statistiche in tempo reale */}
      <footer>
        Axiomas: {Object.values(activeAxioms).filter(Boolean).length}
        Conceptos Validos: {validNodes.size}
      </footer>
    </div>
  );
};
\end{lstlisting}

L'interazione con il Controller avviene tramite la callback \texttt{onChange}: a ogni click sulla checkbox di un assioma, il Boundary invoca \texttt{toggleAxiom(axiom.id)}. Il Controller elabora la richiesta e aggiorna lo stato; Zustand notifica reattivamente il componente, che ri-renderizza l'interfaccia con i nuovi valori di \texttt{activeAxioms} e \texttt{validNodes}. Il footer del pannello mostra in tempo reale il conteggio degli assiomi attivi e dei concetti validi, fornendo un feedback immediato all'utente.

\subsubsection{Complessità Ciclomatica}
È stato analizzato il metodo \texttt{evaluate()} della classe \texttt{Grafo} (\texttt{src/entity/Grafo.ts}), responsabile del ricalcolo topologico e della validazione dei nodi del DAG. Di seguito è riportato il Control Flow Graph (CFG) del metodo.

\begin{figure}[H]
    \centering
    \includegraphics[height=0.45\textheight, keepaspectratio]{images/cfg_diagram.png}
    \caption{Control Flow Graph (CFG) di Grafo.evaluate() con nodi decisionali N1--N7}
\end{figure}

Il codice sorgente del metodo, con i 7 nodi decisionali annotati, è il seguente:

\begin{lstlisting}[language=Java]
evaluate(activeAxioms: Record<string, boolean>): Set<string> {
  const validNodes = new Set<string>();

  for (const [axiomId, isActive] of Object.entries(activeAxioms)) { // N1
    if (isActive)                                                  // N2
      validNodes.add(axiomId);
  }

  for (const nodeId of this.order) {                              // N3
    if (validNodes.has(nodeId)) continue;                         // N4
    const nodo = this.nodi.get(nodeId);
    if (!nodo) continue;                                          // N5
    if (nodo.isAxiom) continue;                                   // N6

    if (nodo.isSatisfiedBy(validNodes))                           // N7
      validNodes.add(nodeId);
  }

  return validNodes;
}
\end{lstlisting}

Il CFG presenta \textbf{7 nodi decisionali}: N1 e N3 sono le condizioni di iterazione dei due cicli \texttt{for}, mentre N2, N4, N5, N6 e N7 sono condizioni \texttt{if} interne. Applicando la formula di McCabe:

\textbf{V(G) = Nodi Decisionali + 1 = 7 + 1 = 8}

Il numero ciclomatico indica che esistono \textbf{8 cammini linearmente indipendenti}. I cammini elementari del CFG sono in realtà 9, ma per la formula di McCabe solo 8 di essi sono linearmente indipendenti: il nono è esprimibile come combinazione lineare degli altri. I 9 cammini sono:

\begin{enumerate}
    \item \textbf{N1(no) $\rightarrow$ N3(no) $\rightarrow$ fine}: nessun assioma, nessun nodo — termina.
    \item \textbf{N1(sì) $\rightarrow$ N2(sì) $\rightarrow$ N1 $\rightarrow$ \ldots}: assioma attivo — aggiunto a \texttt{validNodes}.
    \item \textbf{N1(sì) $\rightarrow$ N2(no) $\rightarrow$ N1 $\rightarrow$ \ldots}: assioma non attivo — ignorato.
    \item \textbf{N1 $\rightarrow$ \ldots $\rightarrow$ N3(no) $\rightarrow$ fine}: ordine topologico vuoto.
    \item \textbf{N1 $\rightarrow$ \ldots $\rightarrow$ N3(sì) $\rightarrow$ N4(sì) $\rightarrow$ N3}: nodo già valido — saltato.
    \item \textbf{N1 $\rightarrow$ \ldots $\rightarrow$ N3(sì) $\rightarrow$ N4(no) $\rightarrow$ N5(no) $\rightarrow$ N3}: nodo assente — ignorato.
    \item \textbf{N1 $\rightarrow$ \ldots $\rightarrow$ N3(sì) $\rightarrow$ N4(no) $\rightarrow$ N5(sì) $\rightarrow$ N6(sì) $\rightarrow$ N3}: è un assioma — saltato.
    \item \textbf{N1 $\rightarrow$ \ldots $\rightarrow$ N3(sì) $\rightarrow$ N4(no) $\rightarrow$ N5(sì) $\rightarrow$ N6(no) $\rightarrow$ N7(sì) $\rightarrow$ N3}: nodo soddisfatto — aggiunto.
    \item \textbf{N1 $\rightarrow$ \ldots $\rightarrow$ N3(sì) $\rightarrow$ N4(no) $\rightarrow$ N5(sì) $\rightarrow$ N6(no) $\rightarrow$ N7(no) $\rightarrow$ N3}: nodo non soddisfatto — ignorato.
\end{enumerate}

Poiché il metodo itera su \textit{molteplici} nodi all'interno di ciascun test case, un singolo test può esercitare più cammini contemporaneamente. Non è quindi necessario un test case per ogni cammino: è sufficiente che ogni cammino elementare venga percorso almeno una volta nell'insieme della suite. \subsection{Vitest - Test di Unità}
Per il test di unità è stato utilizzato \textbf{Vitest}, il framework di testing nativo per progetti Vite/TypeScript. Si riporta di seguito uno dei test case della suite definita nel Capitolo 3, corrispondente a TC\_04 (logica OR tra prove multiple):

\begin{lstlisting}[language=Java]
import { describe, it, expect } from 'vitest';
import { Grafo } from '@/entity/Grafo';

describe('Grafo.evaluate', () => {
  it('handles theorem with multiple proofs (OR logic)', () => {
    const multiProof = {
      topologicalOrder: ['ax-1', 'ax-2', 'def-1', 'def-2', 'thm-1'],
      nodes: {
        'ax-1': { type: 'axioma', proofs: [], directDependencies: [] },
        'ax-2': { type: 'axioma', proofs: [], directDependencies: [] },
        'def-1': { type: 'definicion', proofs: [],
          directDependencies: ['ax-1'] },
        'def-2': { type: 'definicion', proofs: [],
          directDependencies: ['ax-2'] },
        'thm-1': { type: 'teorema',
          proofs: [
            { id: 'p1', dependencies: ['def-1'] },
            { id: 'p2', dependencies: ['def-2'] },
          ],
          directDependencies: [],
        },
      },
    };

    const grafo = Grafo.from(multiProof);
    const result = grafo.evaluate({ 'ax-1': true });
    // Il teorema è valido: la prima prova è soddisfatta
    expect(result.has('thm-1')).toBe(true);

    const resultWithout = grafo.evaluate({});
    // Nessun assioma attivo: il teorema non è valido
    expect(resultWithout.has('thm-1')).toBe(false);
  });
});
\end{lstlisting}

Il test verifica che, in presenza di un teorema con due dimostrazioni alternative, sia sufficiente che una sola di esse abbia tutte le dipendenze soddisfatte perché il teorema risulti valido (logica OR). La suite completa comprende 10 test case che coprono i cammini indipendenti identificati dal CFG.
